# Metodologías Ágiles en Desarrollo de Software

## ¿Qué es Agile?

Agile es una metodología de desarrollo de software que se centra en entregar valor de forma iterativa e incremental. En lugar de planificar todo el proyecto de una sola vez, Agile divide el trabajo en ciclos cortos llamados sprints, permitiendo cambios y mejoras continuas basadas en el feedback de los usuarios.

### Objetivos principales de Agile:

1. **Satisfacción del cliente**: Entregar software funcional regularmente
2. **Flexibilidad**: Aceptar cambios incluso en etapas avanzadas del desarrollo
3. **Colaboración**: Promover la comunicación efectiva entre equipos
4. **Mejora continua**: Aprender de cada iteración y mejorar procesos
5. **Claridad**: Mantener la transparencia en el desarrollo

### Principios clave de Agile:

- Individuos e interacciones sobre procesos y herramientas
- Software funcionando sobre documentación exhaustiva
- Colaboración con el cliente sobre negociación de contratos
- Respuesta ante el cambio sobre seguir un plan

## Scrum

Scrum es un framework ágil específico que proporciona un enfoque estructurado para implementar Agile. Define roles, eventos y artefactos claramente definidos.

### Roles en Scrum:

1. **Product Owner (Dueño del Producto)**
   - Responsable de definir qué se va a construir
   - Gestiona el Product Backlog
   - Prioriza las funcionalidades
   - Comunica las necesidades del cliente

2. **Scrum Master**
   - Facilita que el equipo siga el proceso Scrum
   - Elimina obstáculos que impiden el progreso
   - Capacita al equipo en prácticas ágiles
   - Garantiza que se cumplan las ceremonias

3. **Development Team (Equipo de Desarrollo)**
   - Desarrolla el producto
   - Autoorganizado y multidisciplinario
   - Responsable de la calidad del código
   - Estima el esfuerzo de las tareas

### Conceptos principales:

1. **Sprint**
   - Ciclo de trabajo de 1 a 4 semanas (generalmente 2 semanas)
   - Cantidad fija de trabajo acordada al inicio
   - Objetivo claro demarcado al comienzo

2. **Product Backlog**
   - Lista priorizada de todas las características deseadas
   - Se actualiza constantemente
   - Origen de los elementos del Sprint Backlog

3. **Sprint Backlog**
   - Elementos seleccionados del Product Backlog para el sprint actual
   - Define el trabajo específico a realizar
   - Visible para todo el equipo

4. **Incremento**
   - Producto potencialmente entregable al final de cada sprint
   - Debe ser funcional y estar completo

### Ceremonias de Scrum:

1. **Sprint Planning** - Planificación del sprint
2. **Daily Standup** - Sincronización diaria (15 minutos)
3. **Sprint Review** - Revisión de lo completado
4. **Sprint Retrospective** - Reflexión sobre mejoras

## Kanban

Kanban es un método visual para gestionar el flujo de trabajo. Originario del sistema de producción de Toyota, se centra en limitar el trabajo en progreso (WIP) y visualizar el proceso.

### Principios de Kanban:

1. **Visualizar el trabajo**
   - El trabajo se representa en un tablero con columnas
   - Típicamente: Por Hacer → En Progreso → Completado
   - Permite ver de un vistazo el estado del proyecto

2. **Limitar el trabajo en progreso (WIP)**
   - Cada columna tiene un límite máximo de tareas
   - Evita multitarea y mejora la concentración
   - Mejora la calidad del trabajo

3. **Gestionar el flujo**
   - Enfocarse en terminar trabajo, no en iniciarlo
   - Mover tareas completadas al siguiente estado
   - Evitar cuellos de botella

4. **Hacer políticas explícitas**
   - Definir claramente qué significa cada estado
   - Criterios de entrada y salida de cada columna

5. **Implementar feedback**
   - Revisar regularmente el proceso
   - Ajustar límites de WIP según necesidad

### Ventajas de Kanban:

- **Flexibilidad**: No requiere sprints fijos
- **Continuidad**: Trabajo fluido sin ciclos abruptos
- **Fácil de implementar**: Menos ceremonias que Scrum
- **Visual**: Claridad inmediata del estado

---

## Diferencias principales entre Scrum y Kanban

| Aspecto | Scrum | Kanban |
|--------|-------|--------|
| **Ciclo** | Sprints fijos (2-4 semanas) | Flujo continuo sin ciclos |
| **Cambios** | Solo entre sprints | Cambios en cualquier momento |
| **Roles** | Definidos (PO, SM, Team) | No hay roles específicos |
| **Reuniones** | Ceremonias regulares | Reuniones según sea necesario |
| **WIP Limit** | Flexible por sprint | Límite explícito por columna |
| **Predicibilidad** | Alta (velocidad del equipo) | Variable (según demanda) |

## Cuándo usar cada metodología

### Usar Scrum cuando:

- El proyecto es complejo y requiere cambios frecuentes
- El equipo es grande (5-10+ personas)
- Se necesita estructura y proceso definido
- Los requisitos son inciertos y evolucionan
- Se requiere entrega regular de increments

### Usar Kanban cuando:

- El proyecto es más estable con cambios puntuales
- El equipo prefiere menos ceremonias
- Se necesita máxima flexibilidad
- La demanda es variable y continua
- El trabajo llega en olas o sin patrón

### Usar Scrumban (híbrido):

- Combinar sprints de Scrum con el flujo visual de Kanban
- Mantener ceremonias de Scrum con límites de WIP
- Ideal para equipos en transición o con necesidades mixtas

---

## Conclusión

Ambas metodologías son ágiles y buscan mejorar la entrega de valor al cliente. La elección depende del contexto del proyecto, la naturaleza del equipo y los requisitos específicos. AgileDesk implementa características de ambas, permitiendo a los usuarios gestionar proyectos con la flexibilidad de Kanban y la estructura de Scrum.
